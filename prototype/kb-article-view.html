Troubleshooting Remote VPN Connectivity Drops

1. Overview & Scope

This guide addresses the common issue where end users experience frequent disconnects from the corporate VPN, specifically when utilizing split-tunneling on Windows 11 machines connecting via home ISPs.

2. Resolution Steps

Step 1: Verify the client version.Ensure the user is running the latest version of the VPN client (v4.2 or higher). Older versions have known compatibility issues with recent Windows OS updates. You can verify this by having the user right-click the VPN icon in their system tray, select 'About', and read the version number aloud. If they are on a version older than v4.2, guide them through the standard software update process via the Company Portal.

Step 2: Check MTU Settings.Often, home ISP routers fragment packets if the MTU is too high. Instruct the user or use remote management to lower the VPN adapter MTU to 1350. This is especially prevalent with fiber internet connections using generic ISP-provided gateway devices. You can use PowerShell to make this adjustment quickly: netsh interface ipv4 set subinterface "Ethernet" mtu=1350 store=persistent. Be sure to replace "Ethernet" with the actual name of the VPN virtual adapter.

Pro Tip

Always verify the user's home network signal strength before adjusting MTU settings. Poor Wi-Fi can often mimic VPN instability and cause intermittent drops. A simple speed test can reveal these issues.

Step 3: Update Network Drivers.Check the linked hardware resource in ServiceLink. If the device is a Dell XPS model from 2025, push the latest Intel Wi-Fi driver update via the management console. A known bug in the default factory driver causes intermittent packet loss when dealing with encrypted VPN tunnels. Pushing this update usually takes 2-3 minutes and will briefly disconnect the user from the internet.

3. Advanced MTU Adjustments

If lowering the MTU to 1350 does not fully stabilize the connection, some ISPs may require an even lower MTU. You can instruct users to incrementally lower the MTU by 10 (e.g., 1340, 1330) and test connectivity before finalizing the persistent configuration. Additionally, consider adjusting the TCP MSS clamping if your firewall supports it. Testing with ping sweeps using the -f and -l flags can help determine the exact fragmentation threshold.

4. Escalation Path

If the issue persists after these steps, escalate the ticket to the Level 2 Network team and attach this KB article indicating the steps you have already taken. Please include the user's IP address, the exact time of the disconnects, and any relevant logs from the VPN client to expedite the troubleshooting process.
